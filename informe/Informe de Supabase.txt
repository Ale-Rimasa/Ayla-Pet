Hola,

A partir del 30 de mayo , los nuevos proyectos de Supabase no expondrán las tablas del esquema "public" a la API de datos de forma predeterminada. Cualquier tabla que cree en "public" después de esa fecha requerirá un permiso explícito (GRANT) antes de que supabase-js, PostgREST o GraphQL puedan acceder a ella.

Cómo saber si estás afectado

Hemos detectado 2 proyectos en tu cuenta que utilizan la API de datos.

Te verás afectado si:

Tu aplicación utiliza la API de datos (subase-js, bibliotecas cliente o HTTP directo a "/rest/v1/" o "/graphql/v1/").
Sus scripts de migración o aprovisionamiento crean tablas en "público" sin permisos explícitos.

Usted no se verá afectado si:

Tu aplicación se conecta a Postgres únicamente mediante una cadena de conexión directa (psql, ORM o un servidor de aplicaciones).
Las mesas existentes conservan sus subvenciones actuales. Este cambio se aplica a las mesas nuevas de los proyectos nuevos a partir del 30 de mayo y a todos los proyectos existentes a partir del 30 de octubre.

¿Qué hacer?

Para las nuevas tablas que desee exponer a través de la API de datos, incluya permisos explícitos en el flujo de creación de tablas. Sin un permiso explícito, un rol no tendrá acceso a la tabla creada.

-- Otorgar acceso por rol
otorgar seleccionar
en public.your_table
a anónimo;

otorgar seleccionar, insertar, actualizar, eliminar
en public.your_table
autenticado ;

otorgar seleccionar, insertar, actualizar, eliminar
en public.your_table
al rol de servicio;

-- Habilitar RLS
modificar tabla pública.tu_tabla
habilitar la seguridad a nivel de fila ;

-- Agregar políticas
crear política "los usuarios pueden leer sus propias filas"
en public.your_table
para seleccionar autenticado
usando ( auth.uid () = user_id);
Si falta un permiso, PostgREST devuelve un error "42501" con la instrucción GRANT exacta para solucionarlo.

Fechas de lanzamiento

30 de mayo de 2026: Valor predeterminado para todos los proyectos nuevos.
30 de octubre de 2026 : Se aplica a todos los proyectos existentes.
